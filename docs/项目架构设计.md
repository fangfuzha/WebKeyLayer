# WebKeyLayer 项目架构设计

## 项目概览 & 运行流程

WebKeyLayer 采用 **Rust 后端 + Web 前端 + 素材兼容** 的现代架构：

### 核心运行逻辑

```
1. 启动阶段
   └─ 用户启动 WebKeyLayer 程序
      ├─ 后端启动：Rust 进程启动 Tokio 异步运行时
   │  ├─ 注册全局输入 Hook（键盘与鼠标，Windows 系统级）
      │  ├─ 启动 WebSocket 服务器（0.0.0.0:8080）
      │  └─ 加载配置文件与预设
      └─ UI 启动：系统托盘栏显示图标

2. 运行阶段 - 推流监听
   ├─ 本地：用户监听游戏/直播内容的键盘操作
   │  └─ 全局 Hook 实时捕获所有按键事件 → WebSocket 广播
   ├─ 推流机：在浏览器打开 http://<本机IP>:8080/public/
   │  └─ 接收 WebSocket 事件 → 实时渲染键盘映射画面
   └─ 推流软件：捕获浏览器窗口 → 直播画面中显示按键

3. 配置阶段 - 后台管理
   ├─ 用户双击托盘或右键菜单选择「打开管理面板」
   ├─ 打开本地管理网页（127.0.0.1:8888/admin）
   ├─ 配置主题、样式、布局、预设等选项
   ├─ 配置实时生效（WebSocket 推送给所有客户端）
   └─ 配置保存为 TOML，下次启动自动加载

4. 服务控制 - 托盘菜单
   ├─ 右键「启动/停止服务」：控制键盘监听开关
   ├─ 右键「连接地址」：显示本机 IP:PORT，一键复制给推流机
   ├─ 右键「查看日志」：打开日志文件或查看器
   └─ 右键「退出」：安全关闭程序
```

- **后端**：Rust + Tokio + WebSocket，托盘常驻进程，无重型 GUI 框架
- **前端**：纯 HTML/CSS/JS，推流显示页 (`/public`) + 本地管理页 (`/admin`)
- **素材**：兼容 Input Overlay 预设格式，支持导入/导出/分发

## 核心模块职责分解

### 1. 输入采集层 (`backend/src/keyboard_hook.rs`)

**职责**：全局键盘监听与状态维护

```
┌─────────────────────────────────────┐
│  Windows Global Keyboard Hook       │
│  (System-level event capture)       │
└────────────┬────────────────────────┘
             │ KEY_PRESSED / KEY_RELEASED / MOUSE_BUTTON / MOUSE_MOVE(dx/dy) / MOUSE_WHEEL
             ▼
┌──────────────────────────────────────┐
│ Keyboard State Container            │
│ keyboard_state: HashMap<keycode, bool> │
└────────────┬─────────────────────────┘
             │ publish state change
             ▼
      [Websocket Server]
```

**设计要点**：

- 使用 Rust 系统级 API (Windows: RegisterHotKey 或 SetWindowsHookEx)
- 状态存储为 `HashMap<u16, bool>` (keycode → pressed)
- 线程安全访问（Arc<Mutex<>> 或 atomic）
- Linux 兼容层预留（函数签名相同）

---

### 2. 事件分发层 (`backend/src/websocket_server.rs`)

**职责**：键盘/鼠标事件收集与广播

```
┌──────────────────────────────────────┐
│ Tokio WebSocket Server (0.0.0.0:8080)│
└───┬──────────────────────────────┬───┘
    │                              │
    ▼                              ▼
[推流显示客户端]            [后端管理界面]
(任意局域网设备)           (仅本机 127.0.0.1)
```

**消息格式** (JSON)：统一信封 + 事件负载

```json
{
  "id": "key_pressed_1",
  "type": "key_pressed",
  "timestamp": 1714815600100,
  "payload": {
    "keycode": 16,
    "pressed": true
  }
}
```

**设计要点**：

- Tokio 异步运行时，支持多客户端并发
- 消息队列与背压处理
- 心跳检测与自动重连
- 延迟 < 30ms

---

### 3. 预设兼容层 (`backend/src/preset/importer.rs`) ⭐

**职责**：Input Overlay 格式解析与内部格式转换

```
Input Overlay JSON
       │
       ▼
┌──────────────────────────────┐
│  Importer                    │
│  - Parse JSON                │
│  - Validate fields           │
│  - Map keycodes              │
└───┬────────────┬─────────────┘
    │            │
    ✓            ✗
    │            │
    ▼            ▼
[Internal  [Error Report]
 Format]   - Missing fields
            - Unknown types
            - Validation errors
```

**兼容字段映射**：

| Input Overlay              | WebKeyLayer                | 备注         |
| -------------------------- | -------------------------- | ------------ |
| `type` (ET_KEYBOARD_KEY=1) | `element_type: "keyboard"` | 元素类型转换 |
| `code` (keycode)           | `keycode`                  | 按键码透传   |
| `pos` [x, y]               | `position: {x, y}`         | 页面位置     |
| `mapping` [x, y, w, h]     | `texture: {x, y, w, h}`    | 贴图切片     |
| `id` (e.g. "a")            | `id`                       | 元素标识符   |
| `z_level`                  | `z_index`                  | 堆叠顺序     |

**设计要点**：

- 严格模式：检查所有必需字段，发现缺失时拒绝导入
- 宽松模式：允许部分字段缺失，跳过后继续
- 不兼容项收集后统一报告
- 支持格式版本检查（`version` 字段）

**配置示例** (`config.toml`):

```toml
[theme]
mode = "light"  # "light", "dark", "high_contrast", "system"
primary_color = "#333333"
highlight_color = "#FF5722"

[preset]

[ui]
transparency = 0.9
scale = 1.0

[network]
port = 8080
bind_address = "0.0.0.0"

[i18n]
language = "zh-CN"  # "zh-CN", "en-US"
```

**设计要点**：

- 使用 `toml` crate 解析
- 配置版本号管理
- 文件变化监听（notify crate）
- 热重载不需重启应用

```
                    │
                    │ 广播按键事件
                    │
[推流主机 - 显示端]  [其他设备]
       │                  │
       ▼                  ▼
┌────────────────┐  ┌────────────────┐
│ Browser        │  │ Browser        │
│ /index.html    │  │ /index.html    │
│ (推流捕获)     │  │ (实时监看)     │
└────────────────┘  └────────────────┘
```

### 配置下发流程

```
[用户] → [后端管理页] → [API: POST /api/config]
              │
              ▼
        [配置保存与热重载]
              │
              ▼
        [配置广播给所有客户端]
              │
              ├──► [/index.html] 更新主题/样式
              └──► [/admin] 刷新预览
```

### 预设导入流程

```
[用户上传] → [Importer.parse()]
               │
               ├─ 验证字段 → [兼容性检查]
               │
               ├─ 转换格式 → [内部 Layout Model]
               │
               ├─ 加载资源 → [图片与配置]
               │
               └─ 保存到 config.toml
                   │
                   ▼
              [前端自动刷新预览]
```

## 关键设计决策

| 决策     | 选择             | 理由                      |
| -------- | ---------------- | ------------------------- |
| 后端框架 | Rust + Tokio     | 高性能、低资源、并发安全  |
| GUI 框架 | 轻量级 WebView   | 避免 Tauri 开销，网页化UI |
| 前端构建 | 纯 HTML/CSS/JS   | 无构建依赖，便携性强      |
| 配置格式 | TOML             | 易读易写，热重载友好      |
| 预设格式 | JSON             | 与 Input Overlay 兼容     |
| 网络模型 | WebSocket 广播   | 低延迟、双向通信          |
| 状态管理 | 后端驱动         | 前端无状态，简化同步      |
| 版权管理 | 文件夹级 CREDITS | 每个预设独立声明          |
