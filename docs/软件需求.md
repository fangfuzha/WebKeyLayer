# WebKeyLayer 项目主要需求（支持局域网 / 双机推流 最终版）

## 一、项目概述

WebKeyLayer 是一款**跨平台、局域网通用、支持双机推流的键盘按键可视化浮层工具**。
项目采用 **C/S 架构**：

- **服务端**：运行在**游戏主机/操作主机**，负责全局键盘监听
- **网页前端**：可在**同一局域网内任意设备**打开（推流主机、笔记本、手机等）
- 通过 WebSocket 实现**局域网实时按键同步**
- 推流软件只需捕获浏览器页面，即可实现**无插件、跨设备、双机推流键盘显示**

项目能力对标 Input Overlay，但产品定位与输出目标不同：

- **对标能力**：输入采集、配置驱动布局、按键状态渲染、远程事件分发
- **核心差异**：Input Overlay 以 OBS 插件源输出为主；WebKeyLayer 以网页资源输出为主
- **架构原则**：后端使用 Rust 托盘程序，所有 GUI 均采用网页实现（包括后端管理界面），不依赖 Tauri 等重型桌面 UI 框架

真正实现：**一台电脑操作 → 另一台电脑推流 → 键盘画面实时同步**。

## 二、功能性需求

### 1. 服务端：全局键盘监听（运行在操作主机）

1. 实现 Windows 全局键盘监听，不受窗口焦点影响。
2. 支持字母、数字、符号、方向键、WASD、Ctrl/Shift/Alt/Win 等按键捕获。
3. 按键状态实时上传（按下/抬起）。
4. 服务端启动后**自动在局域网内提供连接**。
5. 显示本机局域网 IP，方便客户端连接。
6. 支持启动/停止监听、退出服务。

---

### 2. 网络通信：支持局域网访问（核心升级点）

1. 服务端 WebSocket 绑定 **0.0.0.0**，允许**同一局域网所有设备连接**。
2. 支持多设备同时连接（可多个设备同时看键盘画面）。
3. 按键事件**低延迟广播**到所有已连接客户端。
4. 客户端断开/重连自动恢复，不影响显示。
5. 仅局域网内通信，**不上公网、不泄露隐私**。

---

### 3. 网页前端：透明键盘浮层（可在推流机打开）

1. 纯网页实现，**支持任意设备访问**：
   - 推流主机（Windows）
   - 笔记本
   - 手机/平板
2. 网页背景**完全透明**，适合直播叠加。
3. 按键按下时高亮，抬起恢复默认样式。
4. 实时映射显示（无需本地持久化）：
   - 打开页面即可看到实时按键映射
   - 所有配置由后端管理
5. 网页自适应大小，可缩放、无边框显示。
6. 支持多语言界面（中文 / 英文）。

---

### 4. 双机推流支持（核心功能）

1. **游戏主机**：运行 WebKeyLayer 服务端 → 监听键盘
2. **推流主机**：打开浏览器访问服务端 → 显示键盘画面 → 推流软件捕获
3. 两台设备在**同一局域网**即可自动连接。
4. 按键延迟 < 30ms，直播无感知同步。
5. 兼容所有推流软件：OBS、剪映直播、VMix、双击推流、各平台直播助手。

### 5. 配置与易用性

1. **后端托盘程序**（Rust + 轻量级本地 WebView 宿主）：
   - 系统托盘栏常驻显示
   - 托盘右键菜单快捷操作：
     - **打开管理面板**：双击托盘或右键「打开管理面板」→ 打开本地管理网页（仅 `127.0.0.1` 可访问）
     - **启动/停止服务**：右键「启动服务」/「停止服务」→ 控制键盘监听与事件广播
     - **显示/复制连接地址**：右键「连接地址」 → 显示当前本机局域网 IP 与端口（如 `192.168.1.100:8080`），一键复制到剪贴板
     - **查看日志**：右键「查看日志」→ 打开日志文件或日志查看面板
     - **退出程序**：右键「退出」→ 安全关闭程序
   - 托盘图标状态指示：运行中/已停止/异常
   - 管理界面采用原生 WebView 容器实现，避免引入额外重型桌面框架
2. **后端管理网页界面**（本地仅可访问）：
   - 映射画面配置与预览：
     - 主题选择（浅色 / 深色 / 高对比度 / 跟随系统）
     - 主题具体定义：
       - 浅色：白底、黑字、亮色高亮
       - 深色：深灰/黑底、白字、彩色高亮
       - 高对比度：纯黑/纯白、超大字体、边框加粗
       - 跟随系统：随 Windows 亮暗主题自动切换
     - 按键样式模板（方形 / 圆形 / 扁平 / 毛玻璃）
     - 样式模板视觉差异：
       - 方形：直角、边框、常规风格
       - 圆形：圆角、柔和边界、现代感
       - 扁平：极简设计、阴影最小
       - 毛玻璃：模糊背景、半透明、高级感
     - 自定义颜色方案：按键颜色、高亮颜色、边框颜色等
     - 键盘布局切换（全键盘 / 精简 WASD）
     - 实时后台预览，无需刷新
     - 页面始终显示键盘映射图标，服务端按下对应按键时切换为按下样式，松开后恢复默认样式
   - 配置管理：
     - 使用 TOML 格式持久化
     - 支持导入/导出配置文件
     - 配置热重载（修改即时生效）
   - 日志查看与诊断：
     - 实时日志面板
     - 连接状态显示（客户端数、延迟、错误统计）
3. 前端配置完全由后端下发，无需本地持久化。
4. 支持中英文界面与语言切换。
5. 无需安装环境，开箱即用，无驱动、无插件。

### 6. 对标 Input Overlay 的模块映射（Web 输出版）

1. **输入采集层（可借鉴 Hook 思路）**：
   - 对标 Input Overlay 的 uiohook / gamepad hook 模块
   - WebKeyLayer 采用 Rust 实现 Windows 全局键盘监听，并预留 Linux 兼容层
2. **状态模型层（可借鉴 input_data 思路）**：
   - 对标 Input Overlay 的 input_data 状态容器（键盘/鼠标/手柄状态）
   - WebKeyLayer 维护统一按键状态表，驱动网页映射图标样式切换（按下/松开）
3. **配置驱动层（可借鉴布局与样式配置思路）**：
   - 对标 Input Overlay 的 overlay/layout 配置化渲染
   - WebKeyLayer 使用 TOML 持久化主题、样式模板、颜色方案与布局，并支持热重载
4. **事件分发层（可借鉴 WebSocket 广播思路）**：
   - 对标 Input Overlay 的 websocket_server 事件广播
   - WebKeyLayer 通过局域网 WebSocket 广播按键事件给全部网页客户端
5. **渲染输出层（关键差异层）**：
   - Input Overlay：渲染目标为 OBS 源
   - WebKeyLayer：渲染目标为网页，供任意可访问网页的推流软件直接捕获
6. **管理界面层（关键实现约束）**：
   - Input Overlay：原生桌面设置界面
   - WebKeyLayer：仅本地管理网页（127.0.0.1），不使用 Tauri，不引入原生桌面 GUI 框架

### 7. 素材格式兼容（Input Overlay 兼容导入）

1. **兼容目标**：
   - 支持导入 Input Overlay 预设素材，至少覆盖常见键盘与鼠标预设
   - 兼容对象包含布局 JSON 与配套贴图资源（如 PNG/JPG/BMP）
2. **兼容字段（第一阶段）**：
   - 顶层字段：`default_width`、`default_height`、`space_h`、`space_v`、`flags`、`overlay_width`、`overlay_height`、`elements`
   - 元素字段：`type`、`pos`、`id`、`z_level`、`mapping`、`code`
3. **元素类型兼容策略**：
   - 第一阶段必须支持：`ET_KEYBOARD_KEY`、`ET_MOUSE_BUTTON`、`ET_TEXTURE`
   - 第二阶段支持：`ET_WHEEL`、`ET_MOUSE_MOVEMENT`、`ET_GAMEPAD_BUTTON`、`ET_ANALOG_STICK`、`ET_TRIGGER`、`ET_DPAD_STICK`、`ET_GAMEPAD_ID`
4. **导入与渲染规则**：
   - 保持 Input Overlay 坐标与贴图切片语义一致（`pos` + `mapping`）
   - 页面常驻显示全部元素，按键按下时切换按下态样式，松开后恢复默认样式
   - 不支持字段或类型时，采用“忽略并告警”策略，不中断整体加载
5. **兼容模式**：
   - 严格模式：仅接受受支持字段与类型，发现不兼容项时提示导入失败原因
   - 宽松模式：允许部分不兼容项，跳过后继续渲染可兼容元素
6. **格式转换能力**：
   - 提供导入后规范化输出能力，将外部素材转换为 WebKeyLayer 内部统一格式
   - 支持将转换结果导出，便于跨设备迁移与版本管理

## 三、非功能性需求

### 1. 局域网双机通信性能

- 按键端到端延迟 < 30ms
- 支持至少 8 个客户端同时稳定连接
- 低带宽占用，不影响游戏与推流

### 2. 兼容性

- **服务端**：Windows 10/11（仅当前支持，保留 Linux 兼容层以便后续扩展）
- **前端**：所有现代浏览器（Chrome/Edge）
- **网络**：局域网（Wi-Fi / 网线均可）
- **后端管理界面**：仅本地访问（127.0.0.1），确保安全
- **桌面承载层**：优先采用轻量级 WebView 方案，保留 Linux 可移植实现空间

### 3. 安全性

- 仅本地/局域网通信，不访问外网
- 不记录按键日志，不上传数据
- 无权限风险，纯用户态运行

### 4. 资源占用

- 服务端内存 < 30MB
- CPU 占用极低，不影响游戏性能

## 五、技术架构

### 后端架构（Rust）

1. **Rust + Tokio + WebSocket + 轻量级 WebView**：
   - Tokio 异步运行时
   - WebSocket 服务负责与前端实时通信
   - 程序形态为托盘常驻进程 + 本地网页管理界面
   - 本地管理界面通过系统原生 WebView 宿主承载，托盘与页面均限制在本机访问
   - 不使用 Tauri 等重型桌面 UI 框架，避免额外资源占用
2. **配置存储**：
   - TOML 格式配置文件
   - 配置文件路径：`AppData/WebKeyLayer/config.toml`
3. **素材兼容导入器**：
   - 负责解析 Input Overlay 预设 JSON 与贴图资源
   - 将外部格式映射为内部统一布局模型
   - 输出不兼容项告警与导入报告
4. **Windows 全局键盘监听**：
   - 系统级别的键盘 Hook
   - 不受窗口焦点影响

### 前端架构（Web）

1. **纯前端实现**：
   - 无状态页面，打开即用
   - 通过 WebSocket 连接后端
   - 前端不存储任何配置
   - 用户可见 GUI 全部为网页（推流显示页 + 本地管理页）
2. **实时渲染**：
   - 接收后端配置下发
   - 接收按键事件实时显示
